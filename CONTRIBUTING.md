# Contribuindo para City App

Obrigado por considerar contribuir para o City App! Este documento fornece diretrizes e instruções para contribuir.

## 🎯 Código de Conduta

Todos os contribuidores devem seguir nosso código de conduta:
- Seja respeitoso
- Aceite críticas construtivas
- Foque no que é melhor para a comunidade
- Mostre empatia com outros membros

## 🐛 Relatando Bugs

Antes de criar um relatório de bug, verifique a lista de issues - você pode descobrir que o bug já foi relatado.

**Como enviar um bom relatório de bug:**

1. Use um título claro e descritivo
2. Descreva os passos exatos para reproduzir o problema
3. Forneça exemplos específicos para demonstrar os passos
4. Descreva o comportamento observado
5. Descreva o comportamento esperado
6. Inclua screenshots se possível
7. Mencione sua versão do Node.js, OS, etc.

## 💡 Sugerindo Melhorias

Sugestões de melhorias são sempre bem-vindas!

**Como enviar uma boa sugestão:**

1. Use um título claro e descritivo
2. Forneça uma descrição detalhada da melhoria sugerida
3. Liste exemplos específicos para demonstrar os passos
4. Descreva o comportamento atual e o comportamento esperado
5. Explique por que essa melhoria seria útil

## 🔄 Pull Requests

- Preencha o template de PR fornecido
- Siga o guia de estilo do projeto
- Inclua screenshots e GIFs animadas em PRs relacionadas à UI
- Termine todos os arquivos com uma nova linha
- Evite plataformas específicas do SO

## 📝 Guia de Estilo

### Git Commit Messages

- Use o tempo presente ("Add feature" não "Added feature")
- Use o modo imperativo ("Move cursor to..." não "Moves cursor to...")
- Limite a primeira linha a 72 caracteres ou menos
- Referencie issues e pull requests livremente após a primeira linha

### JavaScript/TypeScript

- Use 2 espaços para indentação
- Use `const` por padrão, `let` quando necessário
- Use arrow functions quando apropriado
- Use template literals para strings

### React Components

- Use componentes funcionais
- Use hooks para state management
- Nomeie componentes com PascalCase
- Mantenha componentes pequenos e focados

### Tailwind CSS

- Use classes Tailwind ao invés de CSS customizado quando possível
- Siga a ordem: layout → spacing → sizing → colors → effects
- Use responsive prefixes (sm:, md:, lg:, etc.)

## 🚀 Processo de Desenvolvimento

1. Fork o repositório
2. Clone seu fork localmente
3. Crie uma branch para sua feature (`git checkout -b feature/AmazingFeature`)
4. Faça seus commits (`git commit -m 'Add some AmazingFeature'`)
5. Push para a branch (`git push origin feature/AmazingFeature`)
6. Abra um Pull Request
